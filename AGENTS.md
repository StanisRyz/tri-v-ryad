# Agent Guidelines

This is a Godot match-3 battle project intended for Yandex Games. The default layout is vertical 9:16 portrait with a 720x1280 base resolution.

The current stage is a playable battle prototype through Stage 57.1: Symmetric Ice Patterns and Stronger Ice Visuals v0.1 (`IceGenerationRules`/`IcePatternGenerator` now roll a `center_ice_chance` (~0.35 early, ~0.5 medium+) to try a center-anchored `IceShapePreset` shape (`center_square_light`/`center_diamond_light`/`center_square_heavy`/`center_diamond_heavy`, self-symmetric offsets around the exact board center) before falling back to symmetric mirrored blocks (`mirrored_block_2x2`/`2x3`/`3x2`, one copy mirrored across a single random axis) and finally Stage 57's original scattered patterns; active-cell safety/dedup/count-cap/saturation validation and 1-layer/2-layer `frozen_cells` output are unchanged; `TileView`'s placeholder ice overlay is now a strong near-white frost for normal ice and a strong, clearly blue frost for double ice, both far more readable than Stage 57's original faint tints), building on Stage 57 (`ice`-archetype levels generate real frozen cells: `IceGenerationRules.for_tier()`/`IcePatternGenerator` scope ice/double-ice counts per difficulty tier, `BoardChallengeGenerator` routes `ice` to a full active mask plus real `GeneratedBoardChallenge.frozen_cells` seeded from `generation_seed`, applied at battle start through Stage 56's `BoardModel.apply_frozen_cells()`; `normal`/`holes` unchanged, not combined with ice), Stage 56's ice obstacle core (`BoardModel` ice/blocker obstacle layer — `CellObstacleType`, `get_cell_obstacle()`/`set_cell_obstacle()`/`damage_cell_obstacle()`/`clear_cell_obstacle()`/`get_ice_cells()` — separate from tile type/special metadata/active mask; normal ice 1 hit, double ice 2; `IceDamageResolver` damages a directly cleared iced cell and its orthogonal neighbors, deduplicated per clear event, wired into match/cascade/line-special/color-bomb clears and Hammer/Rocket Barrage) and Stage 55.1 (stable per-cell overlay ghosts during animated turns, plus center-hole generation for `holes`-archetype levels on medium+ tiers), with direct app startup on `LevelSelectScreen`, Settings opened from the LevelSelect top panel, GameScreen Menu/Back returning to LevelSelect, inactive/skipped legacy `MainMenuScreen`, damage-only hero abilities frozen behind feature flags, line special tiles, color bombs, a generated 100-level campaign foundation with numbers-only level labels grouped into 10 locked LevelSelect zones, a shared 10-enemy base roster, battle-start random enemy/background/modifier selection, active direct-mode boosters (Hammer, Time Freeze, Rocket Barrage) with selected/used states and strong white affected-cell targeting previews, ImageSlot-backed placeholders, cached image/audio catalogs, saved level completion/stars/unlocks, the Stage 46-49 stepwise board animation pipeline, and Stage 50 victory/defeat result summaries with Next Level, Retry, and Levels actions. `AnimatedTurnFlow` is the active owner of stepwise board visuals during animated turns: swap, match clear, special creation, special activation, gravity/refill, cascade, booster activation/clear, and final board handoff. `TurnFeedbackPresenter` must not replay board movement, match highlights, clear effects, special activation visuals, booster board visuals, refill effects, or whole-board refresh animation after `AnimatedTurnFlow` has handled a valid turn; keep it limited to text/status/enemy feedback for valid animated turns. `GameScreen` is allowed to wire `BattlePresenter`, `BoardView`, `BoardInputController`, `TurnFeedbackPresenter`, `AbilityFeedbackPresenter`, `BoardAnimationController`, `AnimatedTurnFlow`, `BattleEffectController`, `SettingsManager`, `AudioManager`, booster UI selection/preview cleanup, result data preparation, Next Level/Retry/Levels actions, and result-flow reward/completion calls through `ProgressManager`, but the board core, battle core, config layer, progression layer, save layer, settings layer, and asset/audio catalog layers must remain separate from UI implementation details.

## Project Rules

- Use small or medium focused patches.
- Do not implement unrelated systems.
- Use English file and folder names only.
- Do not use spaces or Russian characters in file paths.
- Prefer Control-based UI, containers, and anchors.
- Keep gameplay logic separate from UI.
- Keep battle UI components separate from gameplay logic.
- Keep platform SDK logic separate from gameplay.
- Do not add third-party plugins without explicit request.
- Do not add generated heavy files.
- `GameScreen` may use `BattlePresenter`, but must not directly own board or battle rule details.
